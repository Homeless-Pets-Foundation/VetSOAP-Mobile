# Prevent Stale Recording Upload 404s

## Goal

Make a locally saved recording uploadable even when its remembered server draft was deleted, a request was interrupted, or the app process restarted. The implementation must prevent duplicate visible recordings, preserve local audio until the server proves that the upload or processing result is durable, support one or many audio segments, and remain compatible while the API and mobile releases overlap.

The observed tablet failure is recoverable: the first retry succeeded because the local audio was intact. Historical production telemetry also contains both `presign:HTTP_404` and `confirm:HTTP_404`, so the fix must cover a stale row before presigning and a row that disappears after bytes were uploaded.

This is a coordinated change across:

- `VetSOAP-Connect`: Prisma migration, shared schemas, API routes, R2 verification/retention, tests, deployment, and monitoring.
- `VetSOAP-Mobile`: upload orchestration, durable/stash/draft state propagation, compatibility fallback, recovery UI, tests, and release.

## Safety Invariants

1. Local audio, durable manifests, drafts, and stashes are never deleted merely because preparation, PUT, probing, or confirmation failed.
2. Mobile may delete local audio only after a successful `confirm-upload`, an `already_uploaded` response with server-verified audio, or an `already_processed` response with an existing completed SOAP note.
3. A supplied recording ID is never silently replaced while that row may still exist. Replacement is allowed only after an organization- and user-scoped lookup proves it is absent and the idempotency key does not resolve another canonical row.
4. One stable upload intent maps to one server recording across retries, concurrent requests, process restarts, draft save/resume, and stash round-trips. Starting a genuinely new recording rotates the intent before any server call.
5. All identifiers and metadata lookups remain scoped to both `organizationId` and `userId`; RLS, device binding, MFA, veterinarian, billing, blank-patient-name, and credential gates remain in force.
6. No idempotency key, patient metadata, filename, R2 URL, object key, or raw API error is written to logs, Sentry extras, PostHog properties, or audit metadata. Audit field names and PHI-free outcome codes only.
7. Every returned R2 object key is generated by the server under `recordings/{organizationId}/{recordingId}`. Mobile never supplies or constructs an object key.
8. Recovery is bounded to one stale-ID replacement/re-upload per explicit submit action. Further failure stops, preserves local state, and shows centralized copy that the recording remains saved on the device.

## Server Contract

### Persistent Upload Identity

- Add nullable `Recording.idempotencyKey` (`idempotency_key`, `@db.VarChar(128)`), private nullable `Recording.uploadManifest` (`upload_manifest` JSON), and a unique constraint on `(userId, idempotencyKey)`. Keep both fields null-compatible for historical and non-mobile rows. Manifest version 1 stores only ordered `{ index, fileKey, contentType, fileSizeBytes }` entries; it contains no patient metadata or presigned URLs. Parse database JSON through a strict versioned schema and fail closed on malformed manifests.
- Explicitly omit `idempotencyKey` and `uploadManifest` from `toRecordingResponse()` and every admin/export serializer. The current serializer spreads every Prisma column, so adding either field without this change would expose private upload state to API clients.
- Replace the process-memory 60-second create cache with database-backed idempotency in `POST /api/recordings`. Persist the key in the same callback-style Prisma transaction that creates the row.
- Resolve all keys with `userId` and `organizationId`, even though the database uniqueness key is user-scoped. Catch Prisma `P2002` outside the aborted create transaction, then fetch the winner in a fresh RLS-scoped transaction rather than trying to query through the failed transaction. If the winner is not visible in the caller's organization, fail closed.
- Reject an empty/overlong/malformed key; never truncate it, because truncation can collapse two intents. A draft created on Finish and its later upload preparation use the same key. A historical draft with a null key can be claimed by preparation using a conditional update. A row already carrying a different key returns `409 IDEMPOTENCY_KEY_MISMATCH`.
- Register the static route below before parameterized recording routes so `router.param('id', validateUuidParam)` cannot interpret `prepare-upload` as an ID.

### `POST /api/recordings/prepare-upload`

Protect the route with the same authenticated route stack and recording gates as create plus presign. Before reserving or mutating a row, pass veterinarian, active billing, device/MFA, credential, and blank-patient-name capability checks; confirmation rechecks mutable gates. Use strict shared Zod schemas so unknown properties are rejected.

Required header:

```text
Idempotency-Key: 1..128 visible ASCII characters (0x21..0x7E)
```

Request body:

```ts
{
  existingRecordingId?: string; // UUID
  metadata: {
    patientName: string;         // required current value, max 100
    clientName: string | null;
    species: string | null;
    breed: string | null;
    appointmentType: string | null;
    templateId: string | null;   // UUID when non-null
    foreignLanguage: boolean;
    pimsPatientId: string | null;
  };
  files: Array<{
    fileName: string;            // 1..255, sanitized server-side
    contentType: AllowedAudioMimeType;
    fileSizeBytes: number;       // positive, <= 250 MB
  }>;                            // 1..20, playback order
}
```

Successful response:

```ts
{
  outcome: 'prepared' | 'already_uploaded' | 'already_processed';
  recording: RecordingResponse;
  replacedMissingRecordingId: boolean;
  uploads?: Array<{
    index: number;
    uploadUrl: string;
    fileKey: string;
    expiresAt: string;
  }>;
  warnings: string[];
}
```

- `prepared` always includes `uploads`, with exactly one entry per request file in the same order. `index` must be contiguous from zero.
- `already_uploaded` omits `uploads` and means the database has committed path-confined audio keys and a storage HEAD check proved that every committed object exists. Add an R2 HEAD result that distinguishes `exists`, `not_found`, and service/transport failure; do not treat an outage as a missing object.
- `already_processed` omits `uploads` and means the recording is completed and its SOAP note exists. This remains valid if retained audio has aged out under policy.
- Before returning `already_uploaded` for an `uploaded` row with no saved trigger job, retry the stable idempotent processing enqueue and save its task ID. Apply the same repair to `failed/TRIGGER_FAILED` with verified audio; if enqueue still fails, mark/retain `failed/TRIGGER_FAILED` so mobile enters the existing server-processing retry UI rather than reporting SOAP success.
- The endpoint itself does not return 404. A missing `existingRecordingId` is a recoverable input state; unsupported API deployments produce a route-level 404 before this handler exists.
- Typed failures include `400 INVALID_UPLOAD_PREPARATION`, `409 UPLOAD_INTENT_CONFLICT`, `409 IDEMPOTENCY_KEY_MISMATCH`, `409 RECORDING_METADATA_CONFLICT`, `409 RECORDING_NOT_UPLOADABLE`, `409 SERVER_AUDIO_MISSING`, and `503 STORAGE_UNAVAILABLE`. They never include raw provider errors or presigned URLs.

### Atomic Resolution And Metadata

Perform all database identity and metadata work in a callback-style Prisma transaction under the existing RLS context. Do not call R2 while a database transaction is open.

1. Resolve the idempotency key first. If it finds a row, that row is canonical when `existingRecordingId` is absent, equal to it, or proven missing. If a different supplied row also exists, return `409 UPLOAD_INTENT_CONFLICT` rather than ignoring or replacing either live row.
2. Otherwise, look up `existingRecordingId` with `organizationId` and `userId`. Claim it only if its idempotency key is null or already matches.
3. If neither lookup finds a row, create one in `draft` status with the key. Set `replacedMissingRecordingId=true` only when a supplied ID was proven absent.
4. Classify the row with the ordered status matrix below. A null idempotency key may be claimed on an author-owned historical row as private identity state; only a reusable `draft`, `uploading`, or failed-without-audio row may have recording metadata/status/upload-manifest fields mutated by preparation.
5. For a reusable row, validate `templateId` in the caller's organization, enforce blank-name capability, and apply the complete metadata snapshot, including explicit null clears.
6. Recompute `patientId` in the transaction through `findOrCreateRecordingPatient`, including correction/relink behavior for patient name, client, species, breed, and PIMS identity. Define `pimsPatientId: null` as no submitted PIMS identity; it must not retain a stale link merely because the old draft had one.
7. For a row that is already uploaded/processed, do not mutate metadata during preparation. Load the linked patient's PIMS ID in the response query and compare the request snapshot with the canonical response metadata; return `409 RECORDING_METADATA_CONFLICT` on a material mismatch so mobile preserves its local copy instead of discarding newer edits. Server-enriched patient/client/species/breed/appointment fields may satisfy fields that the submitted snapshot left blank, matching the current mobile assertion behavior; template, language, and nonblank submitted values must match exactly.
8. Do not erase processing history or mutate organization/user ownership. Defer clearing stale upload-only errors until the post-signing compare-and-set transition so signing failure leaves a failed row's diagnostic state intact.
9. Generate deterministic object keys in memory. Single-file key: `recordings/{org}/{recordingId}.{ext}`. Multi-file keys: `recordings/{org}/{recordingId}_segment_{index}.{ext}`. File names affect the sanitized display name, not path ownership. Save the manifest in the same transaction; keep `audioFileSizeBytes` equal to the first file for backward-compatible integer semantics and compute warnings from the request total without persisting an overflowing aggregate.
10. Generate every presigned URL after commit and before returning the response. If any signing call fails, return `503 STORAGE_UNAVAILABLE`; no client has received a URL and no PUT can have started. Leave the canonical row reusable (`draft` stays `draft`, failed stays failed with its diagnostics, and an existing `uploading` row stays `uploading`) so the same key can retry safely.
11. After all URLs are available, conditionally transition a reusable `draft` or failed-without-audio row to `uploading` and clear only upload-attempt error fields. A response interruption after this point is safe because the next request reuses the same row, manifest, and deterministic keys.

The unique constraint is the final concurrency guard. Concurrent preparations with the same key may receive equivalent URLs for the same deterministic keys, but they cannot create two rows. Mobile also suppresses concurrent submit actions for one slot. Confirmation remains compare-and-set so only one request commits audio, and processing enqueue uses its existing stable task idempotency key.

### Status Matrix

Evaluate this table from top to bottom so a completed SOAP note remains authoritative after normal audio retention:

| Canonical row state | Server evidence | Preparation result |
|---|---|---|
| Already uploaded/processed | Material metadata mismatch | `409 RECORDING_METADATA_CONFLICT`; preserve local state |
| `draft` or `uploading` | No committed audio keys | Apply metadata and return `prepared` |
| `failed` | No committed audio keys | Clear upload failure fields, move to `uploading`, and return `prepared` on the same row |
| `completed` | SOAP note exists | Return `already_processed` |
| Any non-completed state | Committed, path-confined keys and all objects verified | Return `already_uploaded`; preserve current processing/failure state |
| Processing/terminal state | No committed audio and no completed SOAP note | `409 RECORDING_NOT_UPLOADABLE`; preserve local audio |
| Any state | Committed key missing from R2 | `409 SERVER_AUDIO_MISSING`; do not replace automatically |
| Any state | R2 verification unavailable | `503 STORAGE_UNAVAILABLE`; preserve local audio |

For `already_uploaded` with `failed`, mobile enters the normal server-side retry/error workflow rather than claiming SOAP creation succeeded. It can still remove local audio because the server object was verified.

### Confirmation Contract

- Move `ConfirmUploadSchema` to the shared schema package, make it strict, and replace prefix-only key checks with exact manifest equality. New manifested uploads accept exactly `recordings/{organizationId}/{recordingId}.{ext}` for one file or contiguous `recordings/{organizationId}/{recordingId}_segment_{index}.{ext}` keys for multiple files. A historical row without a manifest may additionally accept the current mobile legacy form `recordings/{organizationId}/{recordingId}_recording_segment_{index}.{ext}`; anchor every pattern to the complete key so recording-ID prefix collisions fail. Retain segment count/order, status compare-and-set, billing, credential, and blank-name checks.
- Add optional strict `metadata` using the same normalized metadata fields as preparation. The installed mobile build already sends this field, but the current server schema silently strips it.
- Before committing, require submitted keys/count/order to equal the private preparation manifest when one exists. HEAD every submitted key through the tri-state R2 helper and require the expected nonzero size and allowed audio content type. Distinguish missing/mismatched objects (`409 UPLOAD_OBJECT_MISSING`) from storage failure (`503 STORAGE_UNAVAILABLE`). Do not delete local audio on either result. Historical/legacy rows without a manifest use the same exact key grammar and HEAD limits, but cannot require an expected byte-for-byte size.
- After HEAD verification, use one callback transaction to apply metadata/null clears, validate the organization template, relink the patient, and transition `draft|uploading -> uploaded` with committed audio keys. The processing job must receive the committed metadata, and the final response query must include the linked patient fields required for mobile's metadata assertion.
- On a repeated confirm, return the current row as success only when the submitted keys exactly match the committed keys and server evidence satisfies `already_uploaded` or `already_processed`. A 409 with different keys remains an error. If a matching row is still `uploaded` without a saved trigger job, call the idempotent processing enqueue helper again before responding; the stable task key closes the database-commit/process-trigger interruption window.
- Keep `POST /api/recordings`, `POST /api/recordings/:id/upload-url`, and existing confirm behavior backward-compatible for older clients. Their serializers must also omit the persistent idempotency key.

### Orphan Object Retention

A row can disappear after a PUT but before confirmation. Because an object with no database row is invisible to the current cleanup job and a prefix-wide lifecycle rule would also delete referenced failed/in-progress audio, add a paginated scheduled inventory reconciliation. Parse only anchored server-generated `recordings/{org}/{recordingId}...` keys and protect every object whose embedded organization/recording ID still resolves to any recording row, even when legacy pre-confirm rows have no committed key or manifest. For rowless keys, also require age beyond the existing 90-day safety horizon and absence from `audioFileUrl`, `audioSegmentKeys`, private `uploadManifest` entries, and every other recording/document key column. Persist first-seen candidates in a job-only cleanup-candidate table; delete only when the next completed scan repeats all row/key/age checks, then remove the candidate. Run report-only first, cap list/delete work per run, and record PHI-free aggregate counts only. Update the existing completed-recording cleanup to list and delete every anchored object for that recording ID (committed keys plus stale deterministic segment variants) after the 90-day horizon, then atomically clear all audio-key/count columns; deleting only `audioFileUrl` is insufficient. If the R2 token cannot list objects, provision a separately scoped jobs credential rather than broadening the API runtime token. Do not add a client-controlled arbitrary object-delete endpoint.

## Mobile Flow

### Stable Intent And Preflight

- Add a dedicated non-security `uploadIntentId` to `PatientSlot` and every persisted representation. Generate it once before the first Finish-time server draft/create for that recording; timestamp plus `Math.random` is permitted for this non-security identity. Never regenerate it on retry, metadata edit, app restart, stash/resume, or draft restore.
- Use versioned stable keys no longer than 128 characters:
  - Durable capture: `recording-upload-v1:durable:{durable.recordingId}`.
  - Standard slot: `recording-upload-v1:slot:{uploadIntentId}`.
- Add `idempotencyKey` to `createWithSegments()` and propagate `uploadIntentId` through `PatientSlot`, `StashedSlot`, `stashAudioManager`, `useStashedSessions`, `DraftMetadata`, pending draft sync, support-staff recovery vault records, and all restore/replace actions, as required for upload-affecting slot fields. Durable capture continues to use its persisted durable recording ID.
- Rotate `uploadIntentId` only when the user creates a fresh slot or explicitly deletes/starts over after the prior intent's local/server disposition has completed. Reducer resets that merely retry, reopen, or continue a recording retain it. For historical persisted slots without the field, derive `legacy:{slot.id}` once, persist it on the next safe write, and never generate a different migration value on each load.
- Use the same key in the Finish-time draft `create()` call and all later preparation attempts.
- Before any operation that could create/replace a row, prepare URLs, or PUT bytes, preflight every local URI with `getInfoAsync`: local-path validation, existence, nonzero size, allowed content type, per-file 250 MB limit, segment count 1..20, and aggregate progress inputs. Failure leaves server and local state untouched. A complete saved `pendingConfirm` may attempt confirm first because its bytes are already in R2; if stale-row recovery would re-upload, preflight must pass before preparation. The per-slot submitting state disables duplicate presses until that submit action settles.

### Preparation And Upload

1. Build one strict preparation request containing every preflighted file in playback order and the complete normalized current metadata snapshot.
2. Call `prepareUpload()` before any PUT. Validate the response shape with a schema or explicit guards: valid recording UUID, known outcome, `Array.isArray(uploads)`, exact count/order, unique exact-grammar keys, positive expiry, and strings for warnings. For `already_uploaded`/`already_processed`, also verify returned canonical metadata matches the submitted snapshot under the documented server-enrichment rule before treating it as a local-deletion signal.
3. Keep every `validateUploadUrl()` call inside a `try` so phase tagging, timeout cleanup, and local-state preservation still run if validation throws.
4. Replace `onRecordingCreated` with awaited `onRecordingPrepared(recordingId)`, invoked for every successful preparation outcome before branching. Before PUT, update the reducer's `serverDraftId`, `draftStorage.serverDraftId`, and durable manifest using their safe persistence paths; clear `draftMetadataDirty` after `prepared` applied the snapshot or an already outcome proved it matches. If persistence fails, continue only with the database idempotency key as the recovery backstop and capture a PHI-free warning.
5. Clear an obsolete `pendingConfirm` immediately after preparation resolves a different canonical recording. Store the new ID before any PUT and overwrite the pending hint after all PUTs finish.
6. For multi-segment uploads, receive and validate all URLs first, then PUT with the existing bounded concurrency and ordered key array. On the one allowed HTTP 401/403 URL refresh, call `prepare-upload` again with the same key/request, verify that it returns the same canonical row/manifest, and use the refreshed URL at the failed index; never call the legacy per-file route for a manifested upload because its legacy segment key differs. If refresh resolves a different/missing row, cancel and await every active lane before entering the single full stale-row restart. If refresh returns an already outcome, cancel/await lanes and use it only after the normal evidence/metadata checks.
7. Preserve the existing upload timeout, cancellation `.catch()`, progress accounting, and fresh-URL retry behavior. Never fire and forget a Promise.
8. Confirm with the exact ordered server keys and current metadata. Make the all-files-complete callback awaitable. After every segment PUT succeeds, write `pendingConfirm` through the reducer and its available user-scoped persistence before calling confirm: add normalized/cloned support to `DraftMetadata`, preserve it through stash save/resume after a failed submit, and add an atomic local-module durable-manifest setter on Android/iOS when a native manifest exists. Feature-detect that new native method so an older development client degrades to the server backstop instead of throwing. Validate restored recording IDs, counts, and exact key grammar; never store a presigned URL. Never represent a partial multi-segment upload as resumable, and never interpret a pending hint as `confirmedUploadAt`. A hint-write failure is PHI-free telemetry, not an upload failure: continue to confirm because the persistent server intent and deterministic keys remain the correctness backstop.

### Compatibility Fallback

- Deploy the API first. Mobile falls back only when `prepare-upload` itself returns route-level HTTP 404. The new handler must express all input failures as typed non-404 responses.
- In fallback, preflight still runs first. If an existing draft has dirty/current metadata, call `updateDraftMetadata()` before legacy GET/presign; do not rely on the previously ignored confirm metadata. A draft 404 permits fresh create with the same idempotency key; every other metadata/GET failure stops and preserves local audio.
- Validate every legacy presigned URL and retain current single/multi upload behavior. The fallback can be removed only after the minimum supported API deployment is guaranteed.

### Confirmation Recovery

Apply the same policy to an immediate confirm failure and a saved `pendingConfirm` resume:

1. On confirm 404, perform one organization-authenticated `GET /api/recordings/:id` probe.
2. If GET returns `draft` or `uploading`, wait briefly and retry the identical confirmation once; do not re-upload first and do not require local-file preflight for this confirmation-only retry.
3. If GET returns any later state, call preparation with the same key and ID to obtain an `already_uploaded`/`already_processed` proof. A plain GET response is not a local-audio deletion signal.
4. Only if GET itself returns 404, clear the stale pending hint, call preparation with the same stable key and stale `existingRecordingId`, and perform at most one full upload/confirm restart.
5. A probe timeout, network/5xx/401/403/428 response, typed 409, second confirm 404, or second missing-row result stops recovery. Preserve all local audio and the latest useful server ID/hint.

Carry a PHI-free recovery context through the submit operation so success or final failure can be measured. Use PostHog for `upload_stale_recording_recovery` with `stage`, `outcome`, `attempt=1`, `segment_count`, and `mode=durable|standard`. Continue ClientTelemetry phase/error-code reporting for final failures. Do not emit patient data, IDs, keys, filenames, URLs, or raw messages.

Centralize user copy in `src/constants/strings.ts`:

```text
We couldn't finish the upload. The recording is still saved on this device. Check your connection and try again.
```

Do not surface the generic API message "The requested resource was not found" for a recoverable stale recording.

- Security: RLS-enabled route tests, author-only row claim, exact key grammar (including recording-ID prefix collisions), no key/URL/metadata telemetry, and no private-field serializer leak.
- Confirmation: strict metadata accepted and atomically applied, unknown metadata rejected, null clears, patient relink, missing/zero/invalid objects, storage outage, exact repeated-key success, differing-key conflict, new and legacy anchored key grammars, concurrent compare-and-set, idempotent enqueue after a post-commit interruption, job payload metadata, and existing-client payloads.
- R2: HEAD distinguishes missing from outage, every `already_uploaded` object is verified, dry-run inventory pagination/batch caps/two-scan deletion are tested, live-row prefixes remain protected, and completed-prefix/rowless cleanup covers stale segment objects without deleting referenced recoverable audio.
- Run focused tests, `pnpm verify`, migration replay against an empty test database, `pnpm db:check-migrations`, `pnpm rls:check-context`, `pnpm rls:test`, and `pnpm rls:verify`.

### VetSOAP-Mobile

- Preflight precedes every row creation/replacement, preparation/presign, and PUT for single and segmented uploads; missing/empty/oversized/invalid files make none of those requests. Confirmation-only resume remains allowed without a local file.
- Stable keys survive durable manifest, local draft, app restart, stash save/resume, and standard multi-patient slot round-trips; retries retain the intent while a true new/start-over recording rotates it exactly once.
- Preparation responses reject null/malformed arrays, count/order mismatches, duplicate or invalid keys, unknown outcomes, and invalid upload URLs without PUT or local deletion.
- Every segment URL is prepared before the first PUT; ordered keys and bounded concurrency survive out-of-order completion. A manifested URL refresh preserves the exact keys, while compatibility fallback retains the legacy per-file refresh path.
- `onRecordingPrepared` updates reducer, draft storage, and durable manifest before PUT; persistence failure still reuses the database idempotency winner. A complete pending-confirm hint is awaited before confirm where persistent local storage exists, including guarded Android/iOS local-module coverage, and hint-write failure falls back safely to re-preparation/re-upload.
- Cover `prepared`, `already_uploaded`, `already_processed`, unsupported-server fallback, 400/409/503 fail-closed behavior, URL refresh, partial segment failure, and resume paths.
- Cover every confirm probe branch, confirmation-only resume with missing local audio, re-upload refusal when local audio is missing, pending-hint invalidation, one-replacement cap, cancellation/wait of segment lanes, and no local deletion before a server-proven success outcome.
- Update source-contract tests that currently assert the old `onRecordingCreated` and confirm-metadata assumptions; prefer behavior tests for the new orchestration.
- Run focused tests, `npm test`, `npm run typecheck`, `npm run lint`, and `npx expo-doctor`.

### End To End

- Use a staging/test organization and a non-production recording. Never delete a real user's server recording to create the test condition.
- Test on the connected physical tablet because the emulator's silent-audio guard prevents a full upload.
- Create and retain local audio, capture its stable key and draft ID, delete only the test server draft, submit, and verify one automatic replacement reaches transcription and SOAP creation with no user retry and one visible recording.
- Repeat with a forced post-PUT/pre-confirm row deletion to verify the one-time re-upload path and orphan-object retention policy.
- Kill the app after draft creation, after preparation, after PUT, and during confirm; restart each time and verify recovery without duplicate rows or lost local audio.

## Rollout And Monitoring

1. Merge and deploy the database migration, persistent create idempotency, strict confirm metadata support, R2 verification, and backward-compatible preparation endpoint in `VetSOAP-Connect` first. Deploy the changed scheduled cleanup through the repository's jobs deployment workflow separately; an API deploy does not update Trigger.dev jobs.
2. Confirm migration/RLS health, API health, route authentication (a supported route response rather than route-level 404), legacy client create/presign/confirm, serializer omission, orphan-inventory report-only behavior, and production telemetry before mobile release.
3. Implement and release mobile only after the API is live. Before a store build, inspect EAS build/submission history and native version state, choose a marketing version greater than both platforms' recent versions, and update `app.config.ts`, `package.json`, and `package-lock.json`.
4. Run `expo-doctor`; generate the managed Android project locally and run the required production Gradle release build. Run the iOS prebuild/pod/xcodebuild checks on the documented Mac mini. Do not start EAS cloud builds unless separately authorized.
5. Roll out gradually if the stores support it. Keep the route-level-404 fallback during the overlap window.
6. Monitor PHI-free prepare outcomes, recovered-success/failure outcomes, duplicate-key conflicts, R2 verification failures, orphan cleanup, and existing ClientTelemetry upload phases for at least seven days.

Success criteria:

- Zero user-visible stale-row 404 failures in presign or confirm.
- No increase in lost local recordings, duplicate visible recordings, cross-tenant access, metadata mismatch, or R2 object leaks.
- At least 99% of stale-ID recovery attempts that have readable local audio and healthy dependencies reach a server-verified upload without manual retry.
- Legacy clients continue uploading throughout the deployment window.

## Out Of Scope

- Deleting or repairing unrelated production recordings.
- Removing legacy upload endpoints during this change.
- Partial resume of a multi-segment upload; a resumable hint still represents all segments uploaded.
- A client-accessible arbitrary R2 delete API.
- Unrelated untracked files in either repository.
